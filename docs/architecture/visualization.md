# Core Visualization Requirements

## 定位

主通信层的可视化不是附属功能，也不是单纯“把数据画出来”。

它的定位是：

```text
系统级运行监控、调试与观测入口
```

它主要服务于：

* 调试模块之间的通信是否正常
* 查看当前系统运行到了哪一步
* 判断问题出在感知、决策、规划、运动执行中的哪一层
* 查看各模块当前输入输出摘要
* 查看链路延迟、频率、超时、错误
* 支持实验过程中的状态观察与问题定位

也就是说，它更接近一个 runtime observability / debug dashboard。

## 架构位置

当前已经对齐为：

* 可视化不再单独作为独立模块
* 可视化融合在主通信层内部
* 可视化作为主通信层的正式子功能存在

后续架构中应表述为：

> 主通信层负责通信、路由、编排，同时内置可视化与调试监控能力。

## 核心目标

主通信层可视化至少要满足下面几个目标。

### 1. 看模块是否正常运行

要能直观看到：

* 哪些模块在线
* 哪些模块掉线
* 哪些模块异常
* 每个模块最近一次心跳时间
* 当前启用的是哪套模块组合

### 2. 看消息是否正常流动

要能看到：

* 哪些通信链路当前在工作
* 每条链路的消息频率
* 每条链路的平均延迟
* 是否存在超时、堵塞、丢帧
* 最近一条消息什么时候到达

### 3. 看系统当前跑到了哪一步

要能看到整条主链路的当前状态：

```text
Simulation
  -> Perception / Target Estimation
  -> Decision Making
  -> NavDP Path Planning
  -> Locomotion
  -> Simulation
```

每一步最好都能看到：

* 当前输入摘要
* 当前输出摘要
* 最近更新时间
* 当前状态是否有效

### 4. 看空间与任务状态

要能在调试时看到系统当前“理解的世界”是什么样子，包括：

* 机器人当前位置
* 目标估计位置
* 目标预测位置
* Decision Making 输出的子目标
* NavDP 输出的路径
* 当前执行状态

这部分本质上是一个轻量空间态势图。

### 5. 看错误和告警

要能快速暴露系统问题，而不是出错后只能翻日志。

至少要能显示：

* 模块崩溃
* 心跳丢失
* 通信超时
* schema 不匹配
* 数据无效
* 控制链路断裂
* 关键模块输出缺失

## 建议展示内容

下面是当前最合理的一组展示项。

### 1. 模块状态面板

显示每个模块的基本状态，例如：

* 模块名称
* 是否在线
* 最近心跳时间
* 当前状态：`idle` / `running` / `error`
* 当前版本
* 当前容器名或实例名

这里至少应覆盖：

* Simulation
* Perception / Target Estimation
* Decision Making
* NavDP Path Planning
* Locomotion
* Logger / Replay / Evaluation
* 主通信层自身状态

### 2. 通信链路面板

显示主链路和辅助链路的运行情况，例如：

* 链路名称
* 上游模块
* 下游模块
* 当前频率
* 平均延迟
* 最近消息时间
* 是否超时
* 是否异常

重点关注链路包括：

* Simulation ↔ 主通信层
* 主通信层 ↔ Perception
* 主通信层 ↔ Decision Making
* 主通信层 ↔ NavDP
* 主通信层 ↔ Locomotion

### 3. 主流程状态面板

显示当前系统执行链路的阶段状态，例如：

* Simulation 当前状态
* 最近目标估计结果
* 最近决策结果
* 最近路径规划结果
* 最近运动执行结果
* 当前 active control path

这一块的重点不是展示所有细节，而是给出一眼能看懂的“当前系统状态摘要”。

### 4. 机器人状态面板

至少要能展示：

* 机器人位置
* 速度
* 朝向
* 当前模式
* 当前执行状态
* 最近控制命令摘要

如果后面有多机器人，这里要支持按 `robot_id` 查看。

### 5. 目标状态面板

至少要能展示：

* 目标估计位置
* 目标速度估计
* 预测位置
* 置信度
* 当前可见 / 丢失状态
* 最近更新时间

这对调试第一层尤其重要。

### 6. 决策状态面板

至少要能展示：

* Decision Making 当前输出的 subgoal / semantic waypoint
* 角色分配
* 优先级
* 当前任务模式
* 最近一次决策时间

后面即使把 Decision Making 换成 MARL，这一层的可视化也仍然成立。

### 7. 路径规划状态面板

至少要能展示：

* NavDP 当前接收到的中层目标
* 当前输出路径
* waypoint sequence
* 路径是否有效
* 最近规划时间

因为 NavDP 只负责 path planning，所以这里的可视化重点是路径本身，而不是控制动作。

### 8. Locomotion 状态面板

至少要能展示：

* 当前接收到的路径 / 轨迹摘要
* 当前输出的运动执行命令摘要
* 当前运动模式
* 执行是否成功
* 最近执行反馈

### 9. 告警与日志摘要面板

至少要能显示近期重要事件：

* 模块掉线
* 通信超时
* 数据失效
* 控制失败
* 规划失败
* 决策输出无效
* 实验异常结束

## 推荐可视化层次

主通信层可视化可以理解为四层。

### 第一层：系统状态层

看系统是否活着，哪些模块在线，当前跑的是哪个实验。

### 第二层：消息监控层

看消息频率、延迟、超时、丢帧。

### 第三层：任务链路层

看主流程当前执行到哪一步，每一步输出了什么。

### 第四层：空间可视化层

看机器人、目标、路径、子目标等空间信息。

## 实现要求

### 主方式

使用 Web Dashboard。

原因是：

* 容易和 Docker 化系统结合
* 容易做远程访问
* 容易做实时状态展示
* 比单纯依赖仿真器 UI 更适合系统级调试

### 数据推送方式

建议：

* 主通信层内部维护状态总线或状态缓存
* 前端通过 WebSocket 获取实时更新
* 低频查询或筛选操作可以通过 REST 完成

### 与主控制链路的关系

必须注意：

```text
可视化链路不能阻塞主控制链路
```

也就是说：

* dashboard 读取的是状态摘要或镜像数据
* dashboard 不应直接插进控制闭环
* dashboard 异常不应影响主系统运行

## MVP 要求

如果先做第一版，最少建议有下面几块。

### 必须有

* 模块在线状态
* 链路频率与延迟
* 当前目标状态摘要
* 当前决策结果摘要
* 当前路径规划结果摘要
* 当前 locomotion 输出摘要
* 错误和告警列表

### 尽快补上

* 机器人和目标的空间视图
* 路径和子目标显示
* 实验状态显示
* 历史短时回看

## 固定原则

### 原则 1

可视化是主通信层的正式组成部分，不是后补功能。

### 原则 2

可视化的目标是系统调试、观测和问题定位，不只是展示结果。

### 原则 3

可视化要覆盖模块状态、通信状态、主流程状态、空间状态和告警状态。

### 原则 4

可视化应以 Web Dashboard 为主，实时更新以 WebSocket 为主。

### 原则 5

可视化链路必须与主控制闭环解耦，不能影响实时运行。
